SIEM - Detailed Structure and Working Diagram

This document is a detailed companion to README.md.
It uses the full feature set described in the README and turns it into a clear project structure, component layout, folder map, and working diagrams.

1) Full feature set from README.md

Core Functions
- Collect security logs from endpoints, servers, applications, and network devices.
- Accept logs through file upload, HTTP API, Syslog, or agent-based input.
- Normalize different log formats into one common schema.
- Store events for later search and investigation.
- Index events for fast querying.
- Detect suspicious activity using rules and thresholds.
- Generate alerts when rule conditions match.
- Show events, alerts, and trends in a dashboard.
- Support manual investigation and incident tracking.

Security Features
- User authentication.
- Role-based access control.
- Audit logging for user and admin actions.
- Encrypted secrets and protected configuration.
- Raw log preservation for forensic review.

Investigation Features
- Search by time, source, host, user, IP, severity, and tags.
- Filter and sort events quickly.
- Open event details for deeper inspection.
- Link alerts to related events.
- Track cases or incidents manually.

Detection Features
- Rule-based detection.
- Threshold-based detection.
- Repeated-event detection.
- Severity assignment for alerts.
- Alert suppression and deduplication.

Dashboard Features
- Recent alerts view.
- Event volume overview.
- Top sources, hosts, users, and rules.
- Basic trend charts.

Data Features
- Raw event storage.
- Normalized event storage.
- Search index support.
- Tags and metadata support.
- Time-based event records.

Operational Functions
- Log ingestion pipeline.
- Event parsing and normalization.
- Alert creation and storage.
- Search and retrieval.
- Basic reporting and export.

How It Works
- The application continuously receives logs or events from supported sources.
- It parses and normalizes each incoming event.
- It stores raw and normalized data.
- It indexes the event so it can be searched quickly.
- It checks the event against detection rules.
- It creates an alert if a rule matches.
- It updates dashboards and search results for the user.

In simple terms, the SIEM watches incoming security data, organizes it, stores it, searches it, and raises alerts when something suspicious happens.

3) What each folder is responsible for

apps/api
- Receives logs and serves event/alert/case endpoints.

- User interface for dashboards, search, alerts, and cases.

- Stores raw logs, normalized snapshots, and archives.
- Useful for debugging and forensic review.

scripts
- Helper scripts for sample data, imports, and health checks.

prisma
- Database schema and migrations.
- Defines events, alerts, cases, users, and audit records.

4) Detailed system components and their connections

A) Log collection layer
- Sources: endpoints, servers, applications, network devices.
- Inputs: files, Syslog, HTTP API, agent input.
- Output: raw security events.

B) Ingestion layer
- Accepts the event.
- Validates basic shape and source.
- Writes the raw payload to disk.
- Sends the event to the background queue or worker.

C) Normalization layer
- Converts different formats into one canonical event structure.
- Extracts fields such as timestamp, source, host, user, IP, severity, tags, and message.
- Produces normalized records for search and storage.

D) Storage layer
- Stores raw logs for forensic review.
- Stores normalized events for search and reporting.
- Stores alerts and cases for investigation workflow.

E) Search and indexing layer
- Pushes normalized data into a search index.
- Allows search by time, host, user, source, severity, or tags.
- Supports sorting and filtering.

F) Detection layer
- Reads normalized events.
- Runs rule-based and threshold-based checks.
- Detects repeats, suspicious patterns, and severity triggers.
- Creates alerts when a rule is matched.

G) Investigation layer
- Displays event lists and event details.
- Shows alerts, trends, and dashboard summaries.
- Lets analysts create or track cases.

H) Security layer
- Authenticates users.
- Controls roles and permissions.
- Audits important actions.
- Protects secrets and sensitive settings.

5) How data flows inside the application

Step 1: Receive
- A log arrives from a supported source.
- The API receives and validates the request.

Step 2: Store raw event
- The raw payload is saved in data/raw.
- The system keeps it unchanged for future review.

Step 3: Normalize
- A worker reads the raw payload.
- The worker converts it into the canonical event format.

Step 4: Persist
- The normalized event is saved in the database.
- A copy or reference is indexed for search.

Step 5: Detect
- The rule engine checks the event against detection rules.
- Alerts are created if conditions match.

Step 6: Display
- The dashboard and search pages fetch events and alerts.
- Analysts inspect and triage the results.

Step 7: Track and export
- Analysts can link cases, add notes, and export results.

7) Feature-to-component mapping

- Collect logs: Log Source -> Ingestion API
- Accept file upload / HTTP / Syslog / agent input: Ingestion API
- Normalize formats: Worker / Normalizer
- Store events: Database and raw storage
- Index events: Search index
- Detect suspicious activity: Rule engine
- Generate alerts: Alerts store and alert creation service
- Show dashboard: Web UI
- Search and investigation: Search index + API + Web UI
- Authentication and roles: Auth middleware and identity layer
- Audit logging: Audit middleware and audit tables
- Reporting/export: Worker exporter and API export endpoints

9) Practical run model

- Keep the first version single-node.
- Run everything locally on one laptop.
- Use free and open-source services only.
- Store raw payloads on disk.
- Use a worker for background processing.
- Keep the app simple enough to debug manually.
